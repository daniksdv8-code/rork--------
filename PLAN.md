# Продакшен-стабилизация: постоянное хранилище, синхронизация, безопасность

## Контекст

Сейчас все данные хранятся в оперативной памяти сервера (`let store = null`) и теряются при перезапуске. Синхронизация имеет гонки, пароли утекают клиентам, нет защиты API.

> **Примечание по хранилищу:** SQLite через `better-sqlite3` нельзя установить на Rork-платформе (нужна нативная компиляция). Вместо этого будет использован **JSON-файл на диске** с атомарной записью — для одной стоянки (до нескольких тысяч записей) это надёжно и быстро. При самостоятельном деплое на сервер с Bun можно позже заменить на `bun:sqlite`.

## Что будет сделано

### 1. Постоянное хранилище на сервере (критично)
- [x] Все данные сохраняются в JSON-файл на диске при каждом изменении
- [x] При запуске сервера данные загружаются из файла — перезапуск не теряет данные
- [x] Атомарная запись (сначала во временный файл, потом rename) — защита от повреждения при сбое
- [x] Автоматическое создание файла данных при первом запуске

### 2. Первый запуск: админ по умолчанию
- [x] При пустой базе автоматически создаётся пользователь `admin` / `admin`
- [x] После первого входа — сменить пароль через настройки

### 3. Безопасность API
- [x] Пароли **никогда** не отправляются клиентам (полностью убрать из ответов getData)
- [x] Логин-процедура хеширует и сравнивает пароли на сервере, возвращает только безопасные данные
- [x] Все мутации (pushData, resetData, addUser и т.д.) проверяют наличие валидного пользователя

### 4. Синхронизация: сервер — единственный источник истины
- [x] Клиент **всегда** принимает данные с сервера при конфликте версий
- [x] Убрана возможность перезаписи свежих серверных данных старыми локальными
- [x] После restore бэкапа все клиенты получают свежие данные через epoch-механизм (уже есть, будет проверен и усилён)
- [x] Добавлена защита от гонок: если push уже идёт, новые изменения копятся и отправляются после завершения
- [x] При повторном входе приложение подтягивает текущую активную смену с сервера

### 5. Синхронизация пользователей/менеджеров
- [x] Добавление/удаление/блокировка менеджеров идёт **через сервер** и сразу видно на всех устройствах
- [x] Удаление менеджера — soft delete с флагом `deleted: true`, сервер обновляет версию

### 6. Защита смен
- [x] Сервер запрещает открытие второй смены, если уже есть открытая
- [x] При закрытии смены сервер валидирует, что смена действительно открыта

### 7. Мелкие исправления стабильности
- [x] Ограничение размера транзакций/логов на сервере (уже есть, будет проверено)
- [x] Scheduled shifts с soft delete (уже есть, будет проверено)
- [x] Убран мок-данных из initial state (уже чистый `EMPTY_DATA`)
